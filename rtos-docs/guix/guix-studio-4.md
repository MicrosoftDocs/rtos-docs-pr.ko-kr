# <a name="chapter-4-guix-studio-resources"></a>챕터 4: GUIX Studio 리소스

GUIX Studio는 애플리케이션이 색, 글꼴, 픽셀 맵 및 문자열에 사용하는 모든 UI 리소스의 관리를 제공합니다. 다음 섹션에서는 UI 화면 디자인 내에서 리소스를 추가, 수정 및 삭제하는 방법을 설명합니다. 

모든 리소스 관리는 아래 _*_그림 8_**에 표시된 것처럼 GUIX Studio UI의 ***리소스 뷰** _ 내에서 수행됩니다.

![GUIX Studio 리소스 뷰의 스크린샷](./media/guix-studio/image38.jpg)

**그림 8**

## <a name="color-resources"></a>색 리소스

_*_리소스 뷰_*_ 의 ***색** _ 섹션에서 색 리소스를 관리할 수 있습니다. 보기 헤더에서 _ *+* * 필드를 클릭하여 이 보기를 확장할 수 있습니다. 그러면 아래 **_그림 9_** 에 표시된 보기가 보입니다.

![리소스 뷰의 색 섹션의 스크린샷](./media/guix-studio/image_39.png)

**그림 9**

색 리소스는 각각 고유한 논리적 이름을 가진 하나 이상의 색으로 구성됩니다. 예를 들어 **그림 9** 에서 논리적 이름 **CANVAS** 는 화면 배경 채우기 색의 시스템 색 ID이며, 실제 검정색에 연결되어 있습니다. 이 색 리소스는 애플리케이션에서 **GX_COLOR_ID_CANVAS** 를 개체 속성의 색으로 지정할 때마다 사용됩니다.

색 RGB 값을 나타내는 색 "견본"이 왼쪽에 표시되고 그 다음에 색 ID 이름이 표시됩니다. 언제든지 임의의 ID 이름과 연결된 RGB 값을 변경할 수 있습니다. 미리 정의된 시스템 색 ID 이름은 GUIX 라이브러리에서 내부적으로 사용되기 때문에 변경할 수 없습니다. 그러나 색 값은 변경할 수 있습니다. 시스템 색 값을 변경하면 **전체적으로 변경** 됩니다. 즉, 특정 색 할당이 없는 위젯은 새 시스템 색 값을 사용합니다.

테마에 추가한 사용자 지정 색에 대한 색 이름 및 색 값을 모두 변경할 수 있습니다.

색 리소스를 수정하려면 색 리소스에서 두 번 클릭하거나 마우스 오른쪽 단추를 클릭하고 메뉴를 선택합니다. 이 작업을 수행하면 색 정의 대화 상자가 표시됩니다. 이 대화 상자에서 애플리케이션의 UI 요구에 맞게 색 리소스를 수정할 수 있습니다. ***그림 10** _에서는 _ *CANVAS**를 두 번 클릭하는 경우의 수정 대화 상자를 보여줍니다. 이 대화 상자의 모양은 대상 디스플레이의 색 형식 설정에 따라 변경됩니다.

![색 편집 대화 상자의 스크린샷](./media/guix-studio/edit_color.png)

**그림 10**

색 편집 대화 상자의 모양은 현재 디스플레이의 색 농도 및 색 형식 구성에 따라 변경됩니다.

새 색 리소스를 추가하려면 _ *_리소스 뷰_**의 ***색** _ 섹션에서 다음 단추를 선택합니다.

![새 색 추가 단추](./media/guix-studio/image41.jpg)

아래 ***그림 11** 에 표시된 것처럼 결과 색 대화 상자를 사용하여 새 색 리소스를 추가합니다.

![색 편집 대화 상자에 있는 새 색의 스크린샷](./media/guix-studio/new_color.png)

**그림 11**

이러한 단계를 완료하고 **저장** 을 선택한 후에는 실제 녹색인 **NEW_COLOR** 라는 이름의 새 색 리소스를 애플리케이션에서 사용할 수 있습니다.

**디스플레이 색 형식 설정 변경 시 특별 고려 사항:**

새 프로젝트를 만들 때 프로젝트의 디스플레이와 각 디스플레이의 색 형식을 구성하라는 메시지가 자동으로 표시됩니다. 대부분의 경우 한 번 선택하며 이러한 구성 설정을 수정할 필요는 없습니다.

나중에 디스플레이 색 형식 설정을 변경해야 하는 경우를 결정할 수 있습니다. 이 경우 GUIX Studio는 현재 시스템 및 사용자 색 RGB 값을 이전 색 형식에서 새 색 형식에 가장 적합하게 변환합니다. 이러한 변환은 몇 가지 논리적 규칙을 따릅니다.

사용자 정의 색의 경우 GUIX Studio는 항상 이전 색을 새 색 형식에서 가장 일치하는 색으로 변환하려고 합니다. 16bpp 5:6:5 색 형식과 같이 높은 색 농도에서 회색조 또는 단색 형식으로 변환하는 경우 이 변경으로 인해 원치 않는 색 변환이 발생할 수 있습니다. 디스플레이 색 농도 설정이 크게 변경되면 새 사용자 정의 색상표에 대해 일부 수동 업데이트가 필요할 수 있습니다.

미리 정의된 시스템 색의 경우 GUIX Studio는 내부적으로 3개의 고유한 기본 색상표를 정의합니다. 첫 번째 기본 색상표는 4bpp 회색조보다 높은 모든 색 농도에 사용되고, 두 번째 기본 색상표는 회색조 색 형식(1bpp < display_color_format <= 4bpp)에 사용되며, 마지막으로 세 번째 기본 시스템 색상표는 단색 형식에 사용됩니다.

프로젝트 구성 대화 상자를 사용하여 디스플레이 색 형식을 전환하는 경우 다음 규칙이 적용됩니다.

1) 이전 및 새 디스플레이 색 형식이 위에 정의된 대로 다른 기본 시스템 색상표를 사용하는 경우 시스템 색은 각각 미리 정의된 기본 색으로 다시 설정됩니다. 즉, 색을 회색조로 전환하거나 회색조에서 단색 농도로 전환하는 경우 시스템 색은 새 색 농도에 대해 내부적으로 정의된 기본값으로 다시 설정됩니다. 이로 인해 사용자 지정된 색 정보가 일부 손실될 수 있지만, 이 솔루션을 사용하면 디스플레이 색 형식 설정을 크게 변경할 때 합리적인 시작점을 제공합니다.

2) 이전 및 새 색 형식이 동일한 기본 색상표를 사용하는 경우, 즉 색 형식이 크게 변경되지 않은 경우에는 GUIX Studio에서 각 시스템 색을 테스트하여 시스템 색 RGB 값을 기본값에서 수정했는지 확인합니다. 시스템 색 RGB 값이 수정된 경우 GUIX Studio는 이전 색 형식에서 새 색 형식에 가장 일치하도록 변환합니다. 시스템 색이 수정되지 않은 경우 새 색 형식의 기본 색으로 다시 설정됩니다.

**색상표 모드 작업에 대한 특별 고려 사항:**

프로젝트가 256 색상표 모드 색 형식으로 구성된 경우 사용자는 색상표가 설치되고 사용되는 방법을 정의할 수 있습니다. 구성|테마 대화 상자를 사용하여 색상표 정의에 액세스하고 편집할 수 있습니다. 프로젝트가 8bpp로 설정된 경우 "색상표 편집" 단추가 표시됩니다. 이 단추를 클릭하여 색상표 편집 대화 상자를 불러옵니다.

![색상표 편집 대화 상자의 스크린샷](./media/guix-studio/edit_palette.png)

GUIX Studio는 색상표를 "사용자 정의" 섹션과 "자동 생성" 섹션이라는 두 섹션으로 나눕니다. GUIX Studio는 최적의 정교한 색상표 생성 알고리즘을 실행하여 각 테마에 포함된 이미지를 표시하는 데 가장 적합한 색상표를 만듭니다. "미리 정의된 색상표 항목" 필드에 숫자를 입력하여 정의해야 하는 다양한 색상표 항목을 지정하고 이러한 슬롯에 대해 원하는 RGB 값을 입력할 수 있습니다. 나머지 슬롯은 스튜디오에 할당되어 이미지를 표시하기 위한 최적의 색상표를 만듭니다.

이 모드에서 실행할 때 리소스 뷰에서 정의된 색을 편집하려는 경우 색 편집기를 사용하여 사용자가 정의한 미리 정의된 색상표 항목에서만 선택할 수 있습니다. 나머지 색상표 항목은 GUIX Studio에서 자동으로 생성되고, 프로젝트에 추가된 이미지가 수정되면 이에 따라 변경되기 때문입니다.

8bpp 색상표 모드에서 실행할 때 앤티앨리어싱된 글꼴을 표시해야 하는 경우, 각 전경색/배경색에 대한 그라데이션을 만들어 앤티앨리어싱된 텍스트를 표시하는 데 사용할 색상표 항목의 배열을 정의해야 합니다. 각 색 조합의 그라데이션에 대해 8개의 색상표 항목을 사용하거나 더 부드러운 그라데이션을 위해 16개의 색상표 항목을 사용할 수 있습니다. 사용되는 색상표 항목 수는 프로젝트 구성 대화 상자에서 "색상표 모드 앤티앨리어싱된 텍스트 색의 수" 확인란에 따라 결정됩니다. 8개 항목만 포함된 그라데이션을 사용하여 각 색 조합에 사용되는 색상표 항목을 최소화하거나 16개 항목 그라데이션을 사용하여 가장 부드러운 앤티앨리어싱된 텍스트 모양을 제공할 수 있습니다.

앤티앨리어싱된 텍스트를 표시하기 위한 색 그라데이션 정의를 단순화하거나 사용자 고유의 용도에 맞게 색 그라데이션을 생성하기 위해 색상표 편집 대화 상자에서 그라데이션 생성 단추를 제공합니다. 이 기능을 사용하려면 먼저 시작 및 끝 그라데이션 색의 r:g:b 값을 할당해야 합니다.

예를 들어 인덱스 50에서 시작하는 8개의 색상표 항목을 사용하여 중간 회색 배경에 앤티앨리어싱된 빨간색 텍스트를 표시하려면 r:g:b 값 255:0:0을 색상표 인덱스 50에 할당하고 r:g:b 값 128:128:128을 색상표 인덱스 57에 할당합니다. 이 대화 상자의 시작 인덱스 및 끝 인덱스 필드에 이러한 색상표 인덱스 값을 입력하고 그라데이션 생성 단추를 클릭합니다. 51에서 56까지의 색상표 항목은 경계 색 간의 부드러운 그라데이션 전환을 정의하도록 초기화됩니다.

## <a name="font-resources"></a>글꼴 리소스

글꼴 리소스를 관리하려면 _ *+* * 필드를 클릭하여 먼저 _*_리소스 뷰_*_ 의 ***글꼴** _ 섹션을 확장해야 합니다. 그러면 아래 **_그림 12_** 에 표시된 대화 상자가 보입니다.

![리소스 뷰에 있는 글꼴 섹션의 스크린샷](./media/guix-studio/image44.jpg)

**그림 12**

글꼴 리소스는 각각 고유한 논리적 이름을 가진 하나 이상의 글꼴로 구성됩니다. 예를 들어 **그림 12** 에서 논리적 이름 **SYSTEM** 은 특정 글꼴과 연결됩니다. 이 글꼴 리소스는 애플리케이션에서 **SYSTEM** 을 개체 속성의 글꼴로 지정할 때마다 사용됩니다. 글꼴 그룹은 왼쪽의 글꼴 문자 모양에 대한 WYSIWYG 미리 보기, 픽셀 단위의 글꼴 높이, 글꼴 ID 이름 및 글꼴 크기(KB)를 표시합니다.

위의 보기에서 처음 4개 글꼴은 GUIX 라이브러리에 필요한 미리 정의된 기본 글꼴입니다. 이러한 글꼴에 연결된 글꼴 데이터를 변경할 수 있지만 이러한 글꼴 ID 이름을 변경할 수는 없습니다.

위에 표시된 마지막 글꼴("기울임꼴")은 사용자가 프로젝트에 추가한 사용자 지정 글꼴입니다.

글꼴 리소스를 수정하려면 글꼴 리소스에서 두 번 클릭하거나 마우스 오른쪽 단추를 클릭하고 메뉴를 선택합니다. 이 대화 상자에서 애플리케이션의 UI 요구에 맞게 글꼴 리소스를 수정할 수 있습니다. ***그림 13** _에서는 _ *SYSTEM**을 두 번 클릭하는 경우의 수정 대화 상자를 보여줍니다.

![[SYSTEM을 두 번 클릭하는 경우의 수정 대화 상자 스크린샷](./media/guix-studio/edit_system_font.png)

**그림 13**

새 글꼴 리소스를 추가하려면 _ *_리소스 뷰_**의 ***글꼴** _ 섹션에서 다음 단추를 선택합니다.

![새 글꼴 추가 단추](./media/guix-studio/image46.jpg)

이렇게 하면 아래 ***그림 14*** 에 표시된 것처럼 `Font Edit` 대화 상자를 호출하여 새 글꼴 리소스를 추가합니다.

![새 글꼴 리소스를 추가하는 수정 대화 상자의 스크린샷](./media/guix-studio/add_new_font.png)

**그림 14**

새 GUIX 글꼴은 GUIX Studio에서 선택된 트루타입 글꼴을 특정 크기로 렌더링하여 만듭니다. 따라서 위의 대화 상자에는 먼저 트루타입 글꼴 경로가 필요합니다. 찾아보기 단추를 사용하여 개발 시스템에서 글꼴 파일이 포함된 디렉터리를 찾을 수 있습니다. GUIX Studio를 설치한 위치에 관계 없이 GUIX/fonts 하위 폴더에도 여러 가지 트루타입 글꼴이 포함되어 있습니다.

가능하면 트루타입 글꼴 파일의 위치는 프로젝트 상대 경로를 사용하여 내부적으로 저장됩니다. 이러한 이유로 GUIX Studio 프로젝트를 하나의 개발 스테이션에서 다른 개발 스테이션으로 이동할 수 있도록 모든 글꼴 파일을 공통 위치에 유지하고 프로젝트 및 글꼴 파일에 대해 공통 디렉터리 트리 구조를 사용하는 것이 중요합니다.

글꼴 이름 필드를 사용하여 글꼴 리소스 ID 이름을 지정할 수 있습니다. GUIX Studio에서 생성된 코드에 사용되고, 글꼴을 참조할 때 애플리케이션에서도 사용되는 리소스 ID입니다. 이 이름은 C 변수 명명 구문 요구 사항을 따라야 합니다.

입력으로 사용할 트루타입 글꼴 파일을 선택했으면 글꼴 논리적 이름을 입력합니다.

"**커닝 정보 생성**" 확인란은 GUIX Studio가 커닝 정보를 생성된 글꼴 내에 포함하도록 지시합니다. 이는 문자열에서 연속 문자 모양의 상대 위치를 조정하는 데 사용됩니다. 문자열에 커닝을 적용하려면 커닝 정보를 포함하는 글꼴을 사용하고 이 확인란을 설정해야 합니다. 커닝 정보를 사용하여 텍스트를 렌더링하는 것을 지원하기 위해 GUIX 라이브러리 빌드 옵션인 "GX_FONT_KERNING_SUPPORT"도 정의해야 합니다.

"**문자열 테이블에 의해 정의된 문자 집합 포함**" 확인란은 GUIX Studio가 정적 문자열 테이블에서 참조하는 문자 모양을 생성된 글꼴 내에 포함하도록 지시합니다. 아래에 나열된 문자 범위를 선택하고 편집하여 추가 문자 모양을 포함할 수 있지만, 이 옵션을 선택하여 문자열 테이블 내에서 정의된 문자열을 표시하는 데 필요한 최소 문자 집합을 신속하게 생성할 수 있습니다. 물론, 문자열 테이블이 트루타입 원본 글꼴에 없는 문자 모양을 사용하는 경우 해당 문자는 GUIX 글꼴에서 사용할 수 없으며 대상 시스템에 표시되지 않습니다.

정적으로 정의된 문자열 테이블 내에서 사용할 수 없는 문자를 포함하는 글꼴 또는 전체 글꼴을 생성하려면 아래 목록에서 문자 범위를 선택할 수도 있습니다. 원하는 수의 문자 범위를 선택할 수 있으며, 선택한 각 범위 내에 포함될 실제 시작 및 끝 문자 코드를 편집할 수 있습니다.

미리 정의된 문자 범위와 페이지 이름에는 현재 사용 중인 활성 언어에 필요한 문자 집합을 쉽게 선택할 수 있는 제안만 있습니다. 나열된 언어 이름은 생성된 GUIX 글꼴에 영향을 주지 않으며, 사용하거나 선택한 문자 범위에 대해 원하는 모든 16진수 문자 범위를 자유롭게 입력할 수 있습니다.

예를 들어 숫자만 포함된 글꼴을 생성하려는 경우, "Ascii" 코드 페이지를 선택하고 시작 값 0030과 끝 값 0039를 입력하여 숫자만 포함된 글꼴을 생성할 수 있습니다. 유니코드 문자 테이블의 일반적인 표기법인 16진수로 인코딩된 문자 범위 값을 확인합니다.

기본적으로 GUIX Studio 및 GUIX 라이브러리는 현재 사용 중인 모든 활성 언어, 수학 양식 및 기타 기호를 포함하는 0x0000에서 0xffff까지의 문자 코드를 지원합니다. 특정 사용자 지정 영역을 포함하여 0xffff 값 위의 문자 코드를 사용해야 하는 경우 "확장 문자 범위 지원" 확인란을 설정해야 합니다. 이 확인란을 선택하면 GUIX Studio에서 사용자가 유니코드 사용자 지정 문자 범위를 포함하는 0x0000에서 0x10ffff까지의 문자 범위를 지정할 수 있습니다. 이 확장 문자 범위가 필요한 경우 GUIX 라이브러리에서 16비트 문자 코드를 지원하는 기본 구성 대신 32비트 문자 코드를 내부적으로 지원하도록 GUIX 라이브러리 빌드 옵션인 "GX_EXTENDED_UNICODE_SUPPORT"도 정의해야 합니다.

"문자열 테이블에 의해 정의된 문자 집합 포함" 확인란 및 아래 목록에 있는 하나 이상의 문자 범위를 모두 선택하는 경우, GUIX Studio에서는 이러한 선택 항목을 선택된 범위와 문자열 테이블 내에서 사용된 문자의 상위 집합으로 결합합니다. 물론 GUIX Studio에서 요청된 각 문자 값에 대한 의미 있는 문자 모양을 생성하기 위해, 선택한 트루타입 원본 글꼴에도 필요한 문자가 포함되어야 합니다.

문자 범위가 결정되면 픽셀 단위의 글꼴 높이 및 글꼴 형식을 지정합니다. 앤티앨리어싱된 글꼴과 이진 글꼴이 모두 지원됩니다. 이진 글꼴에는 정적 데이터 스토리지 영역이 더 필요하지 않지만, 앤티앨리어싱된 글꼴은 4bpp 회색조 이상 색 농도에서 실행되는 대상에 가장 적합한 모양을 생성합니다.

>[!NOTE]  
> *"글꼴 높이"는 글꼴의 EM 사각형을 나타냅니다. 기존 금속 활자에서 EM 사각형은 각 문자가 위로 올라가는 금속 몸체의 선 높이와 같으며 각 금속 몸체의 크기와 동일합니다. 금속 활자에서 문자의 물리적 크기는 일반적으로 EM 사각형을 초과할 수 없습니다. 디지털 형식에서 EM은 디지털 글꼴의 디자인 공간으로 사용되는 임의의 해상도 그리드입니다. 이러한 디지털 글꼴의 경우 위 첨자 및 아래 첨자와 같은 일부 문자 모양 기능이 EM 사각형의 한도를 초과하여 확장될 수 있습니다. 최종 결과는 특정 글꼴을 완전히 표시하는 데 필요한 위젯 높이가, 요청된 글꼴 픽셀 높이보다 약간 커야 하는 경우가 많습니다.*

모든 글꼴 구성 필드가 완료되면 확인 단추를 클릭하여 새 글꼴 리소스를 만듭니다. GUIX Studio는 선택한 속성을 사용하여 GUIX 호환 글꼴을 생성하고, 해당 글꼴을 프로젝트 리소스에 추가하고, 글꼴을 애플리케이션에서 사용할 수 있도록 합니다.

## <a name="pixel-map-resources"></a>픽셀 맵 리소스

픽셀 맵 리소스를 관리하려면 _ *+* * 필드를 클릭하여 먼저 _*_리소스 뷰_*_ 의 ***픽셀 맵** _ 섹션을 확장해야 합니다. 그러면 아래 **_그림 15_** 에 표시된 대화 상자가 보입니다.

`Pixelmap` 그룹이 확장되면 다음과 비슷한 미리 보기가 표시됩니다.

![리소스 뷰에 있는 픽셀 맵 섹션의 스크린샷](./media/guix-studio/pixelmap_view.png)

**그림 15**

픽셀 맵 리소스는 하나 이상의 픽셀 맵으로 구성되며, 각 픽셀 맵은 왼쪽의 픽셀 맵 이미지 미리 보기, 픽셀 단위의 픽셀 맵 크기, 출력 리소스 파일의 고유한 논리적 이름과 픽셀 맵 스토리지 크기(KB)로 구성됩니다.

픽셀 맵의 첫 번째 그룹은 라디오 단추 및 확인란과 같은 GUIX 위젯에 필요한 미리 정의된 시스템 픽셀 맵으로 구성됩니다. 시스템 픽셀 맵과 연결된 픽셀 맵 데이터를 변경할 수 있지만 이러한 픽셀 맵 ID 이름은 변경할 수 없습니다. 또한 위에 표시된 것은 "BACKGROUND" 및 "BUTTON_ACTIVE"와 같은 이름을 가진 여러 사용자 지정 픽셀 맵입니다. 이는 GX_PIXELMAP_BUTTON 위젯을 렌더링하는 데 사용될 수 있는 프로젝트에 사용자가 추가한 픽셀 맵의 예입니다.

많은 프로젝트에는 많은 수의 픽셀 맵이 포함되어 있으므로 픽셀 맵 보기를 사용하여 픽셀 맵 이미지를 구성하는 데 필요한 수의 픽셀 맵 폴더를 정의할 수 있습니다. 

***리소스 뷰*** 의 `Pixelmaps` 섹션 헤더를 마우스 오른쪽 단추로 클릭하고 "폴더 추가"를 선택하면 새 픽셀 맵 폴더가 추가됩니다.

픽셀 맵 리소스를 수정하려면 픽셀 맵 리소스에서 두 번 클릭하거나 마우스 오른쪽 단추를 클릭하고 메뉴를 선택합니다. 이 대화 상자에서 애플리케이션의 UI 요구에 맞게 픽셀 맵 리소스를 수정할 수 있습니다. ***그림 16** _에서는 _ *RADIO_ON**을 두 번 클릭하는 경우의 수정 대화 상자를 보여줍니다.

![픽셀 맵 편집 대화 상자의 스크린샷](./media/guix-studio/image49.jpg)

**그림 16**

`Edit Pixelmap` 대화 상자를 사용하여 새 픽셀 맵을 정의하거나 기존 픽셀 맵의 내용을 수정할 수 있습니다. GUIX Studio는 내부적으로 입력 이미지를 읽고 이미지를 GUIX 라이브러리에서 사용할 수 있는 `GUIX GX_PIXELMAP` 형식으로 변환합니다. 또한 GUIX Studio는 들어오는 이미지의 색 공간을 이 픽셀 맵이 사용될 디스플레이의 색 공간으로 변환합니다.

이 대화 상자의 첫 번째 필드는 원본 이미지에 대한 경로입니다. GUIX Studio는 PNG(.png) 또는 JPEG(.jpg) 형식 이미지 파일의 입력을 지원합니다. "찾아보기" 단추를 사용하여 로컬 파일 시스템에서 원하는 입력 파일을 찾을 수 있습니다.

가능하면 입력 이미지 파일의 위치는 프로젝트 상대 경로를 사용하여 내부적으로 저장됩니다. 이러한 이유로 GUIX Studio 프로젝트를 하나의 개발 스테이션에서 다른 개발 스테이션으로 이동하고 입력 이미지 데이터를 추적할 수 있도록 모든 이미지 파일을 공통 위치에 유지하고 프로젝트 및 이미지 파일에 대해 공통 디렉터리 트리 구조를 사용하는 것이 중요합니다.

`Pixelmap ID` 필드를 사용하여 픽셀 맵 리소스의 논리적 이름을 지정할 수 있습니다. 여기서 입력하는 이름은 고유해야 하며 C 변수 명명 구문 규칙을 따라야 합니다.

출력 파일 지정 확인란을 사용하여 각 픽셀 맵에 고유한 출력 파일을 지정할 수 있습니다. 이 확인란을 선택하지 않으면 픽셀 맵 데이터가 이 디스플레이의 기본 리소스 파일에 기록됩니다. 이 확인란을 선택하면 이 픽셀 맵의 데이터가 기록될 특정 파일 이름을 입력할 수 있습니다. 이 옵션은 C 배열이 매우 클 수 있는 픽셀 맵 데이터를 여러 출력 파일로 나눌 수 있도록 하기 위한 것입니다. 특정 컴파일러는 수십만 개의 소스 줄이 있는 C 파일을 처리하려고 합니다.

"출력 압축" 확인란을 사용하면 픽셀 맵 출력이 독점적인 GUIX 압축 알고리즘을 사용하는지 여부를 지정할 수 있습니다. 압축된 출력 파일은 일반적으로 작지만 대상에서 렌더링하는 데에도 프로세서 시간이 필요합니다. 대부분의 경우 큰 픽셀 맵에는 압축을 선택하고 작은 픽셀 맵에는 비압축 형식을 사용합니다.

이 `Include Alpha Channel` 확인란은 GUIX Studio에서 때때로 .png 형식의 입력 파일에 있는 알파 채널 정보를 활용하는 방법을 결정합니다. 이 확인란을 선택하고 디스플레이를 16bpp 색 농도 이상으로 실행하는 경우 GUIX Studio는 전체 들어오는 알파 데이터를 출력 파일에 보존합니다. 이 확인란을 선택하지 않으면 GUIX는 약간 작은 출력 파일을 생성합니다. 이 출력 파일에는 투명도가 포함될 수 있지만 전체 알파 혼합 정보는 포함되지 않습니다.

이 `Dither` 확인란을 선택하면 GUIX Studio에서 더 낮은 색 농도 표시에 사용할 입력 이미지를 변환할 때 고급 디더링 알고리즘을 적용하도록 지시합니다. 일반적으로 디더링을 사용하도록 설정되지만 압축을 사용하는 경우 반복 픽셀이 더 적기 때문에 출력 파일이 더 커질 수 있습니다.

모든 옵션이 원하는 대로 설정되면 확인 단추를 클릭하여 새 픽셀 맵 리소스를 생성합니다. GUIX Studio는 입력 이미지 파일을 읽고, 압축을 풀고, 색 공간 변환과 디더링을 수행하고, 선택적으로 데이터를 다시 압축하고, 데이터를 GUIX 호환 `GX_PIXELMAP` 형식으로 저장합니다. 새 픽셀 맵이 프로젝트 리소스에 추가되고 애플리케이션에서 사용할 수 있게 됩니다.

새 픽셀 맵 리소스를 추가하려면 ***리소스 뷰*** 의 `Pixelmaps` 섹션 내에서 다음 단추를 선택합니다.

![새 픽셀 맵 추가 단추](./media/guix-studio/image50.jpg)

## <a name="string-resources"></a>문자열 리소스

문자열 그룹이 확장되면 아래와 같이 프로젝트 문자열 테이블의 미리 보기가 표시됩니다.

![확장된 문자열 그룹의 스크린샷](./media/guix-studio/string_res_view.png)

**그림 17**

문자열 리소스는 각각 고유한 논리적 이름을 가진 하나 이상의 문자열로 구성됩니다. 예를 들어 **그림 17** 에서 논리적 이름 "PATIENT_LIST"는 오른쪽에 표시된 문자열 "Patient List"와 연결됩니다. 이 문자열 리소스는 애플리케이션에서 PATIENT_LIST를 개체 속성의 문자열로 지정할 때마다 사용됩니다.

모든 리소스 종류에 대한 ID 이름은 항상 C 구문 호환 변수 이름이어야 합니다. 이러한 이름은 프로젝트 리소스 파일 및 사양 파일이 스튜디오에서 생성될 때 광범위하게 사용됩니다.

문자열 리소스를 수정하려면 문자열 리소스에서 두 번 클릭하거나 마우스 오른쪽 단추를 클릭하고 메뉴를 선택하여 ***문자열 테이블 편집기** _ 대화 상자를 호출합니다. _*_문자열 테이블 편집기_*_ 대화 상자에서 애플리케이션의 UI 요구에 맞게 문자열 리소스를 수정할 수 있습니다. _*_그림 18_*_ 에서는 _ *STRING_13**을 두 번 클릭하는 경우의 수정 대화 상자를 보여줍니다.

이 경우 문자열 ID 이름이 왼쪽에 표시되며, 첫 번째 또는 참조 언어의 문자열 내용이 오른쪽에 표시됩니다. 물론 정확한 문자열 내용은 애플리케이션에만 표시되지만 문자열 그룹 미리 보기의 레이아웃은 일관됩니다.

GUIX Studio는 문자열 테이블을 정의하고 유지 관리하여 정적 텍스트 및 다국어 애플리케이션을 지원합니다. 문자열 테이블은 각 레코드에 대해 하나의 문자열 ID를 정의하고, 지원되는 각 언어의 각 레코드에 대해 하나의 문자열 상수를 정의합니다.

애플리케이션에서 지원되는 언어는 언어 구성 대화 상자를 사용하여 정의되며 여기에 표시됩니다.

![언어 구성 대화 상자의 스크린샷](./media/guix-studio/config_languages.png)

**그림 18**

언어 구성 대화 상자는 애플리케이션 메뉴에서 구성 | 언어 명령을 사용하여 호출됩니다. 이 대화 상자를 사용하여 애플리케이션에서 지원할 언어 수와 각 언어에 연결할 이름 또는 언어 ID를 정의할 수 있습니다. 프로젝트를 만든 후에 지원되는 언어를 수정할 수 있지만, 언어를 제거하는 경우 해당 언어와 연결된 문자열 데이터도 제거되고 검색할 수 없다는 것을 알고 있어야 합니다.

"**정적으로 정의됨**" 확인란은 선택한 언어가 생성된 리소스 파일의 소스 코드 형식에서 정적으로 정의됨을 나타냅니다. 정적으로 정의된 언어가 없는 경우 언어 테이블 포인터는 생성된 표시 테이블에서 NULL로 설정되고 GUIX 라이브러리에서 제공하는 이진 리소스 로더 API를 사용하여 애플리케이션에서 언어를 로드하고 설치해야 합니다.

"**양방향 텍스트 지원**" 확인란은 GUIX Studio가 양방향 텍스트 렌더링을 지원하도록 지시합니다. 이 언어로 입력할 문자열에 양방향 텍스트 렌더링이 필요한 경우 이 확인란을 설정해야 합니다.

"**표시 순서대로 양방향 텍스트 생성**" 확인란은 GUIX Studio가 표시 순서대로 출력 파일에 양방향 텍스트를 생성하도록 지시합니다. 이 옵션을 선택하는 경우 양방향 텍스트를 올바르게 렌더링하기 위해 GUIX 라이브러리 내에서 런타임 처리가 필요하지 않습니다. 이 옵션을 선택하면 GUIX 라이브러리 내에서 양방향 텍스트 렌더링을 사용하도록 설정하면 안 됩니다. 이 구성은 최상의 런타임 성능을 제공하지만 동적으로 정의된 양방향 텍스트 문자열의 렌더링은 지원하지 않습니다.

첫 번째 언어 또는 "인덱스 1" 언어를 "참조 언어"라고 합니다. 이 언어는 UI 디자인을 정의하고 편집할 때 GUIX Studio에서 사용하는 언어입니다. 문자열 테이블의 다른 모든 언어는 번역 언어라고 합니다. GUIX Studio는 업계 표준 XLIFF 또는 CSV 형식 데이터 파일의 문자열 테이블 데이터를 내보내고 가져오는 것을 지원하므로, 애플리케이션 개발자가 참조 언어 이외의 지원되는 언어에 대한 번역을 추가하는 데 도움을 줄 수 있는 번역가와 문자열 정보를 교환하는 데 편리합니다. GUIX 문자열 테이블을 XLIFF 또는 CSV 파일로 내보내면 참조 언어가 한 번역 언어와 함께 XLIFF 또는 CSV 문자열 데이터 교환 파일에 포함됩니다. 마찬가지로, XLIFF 또는 CSV 파일을 가져올 때 가져온 데이터를 사용하여 GUIX 문자열 테이블에서 하나의 번역 언어를 채웁니다.

![문자열 테이블 편집기의 스크린샷](./media/guix-studio/image53.jpg)

**그림 19**

문자열 테이블 편집기 대화 상자는 먼저 왼쪽에 문자열 ID의 목록을 표시한 다음 참조 언어 문자열 데이터를 표시합니다. 둘 이상의 언어가 정의된 경우 세 번째 열에는 지원되는 번역 언어 중 하나가 표시됩니다. 참조 언어 열의 오른쪽 위에 있는 작은 화살표를 클릭하여 세 번째 열을 열고 닫을 수 있습니다.

번역 언어 열이 표시되면 문자열 목록에서 번역 언어 열의 오른쪽 위에 있는 작은 화살표를 클릭하여 프로젝트에 포함된 번역 언어를 순환할 수 있습니다.

테이블에서 선택할 레코드를 클릭하여 문자열 레코드를 편집할 수 있습니다. 레코드를 선택하면 레코드 문자열 ID와 문자열 내용이 테이블 뷰 아래의 필드에 표시됩니다. 이러한 필드에 새 값을 입력하여 문자열 ID 및 문자열 내용을 수정할 수 있습니다.

테이블 뷰의 오른쪽에 있는 상자에는 선택한 문자열을 참조하는 위젯의 미리 보기가 표시됩니다. 이는 편집된 문자열이 특정 위젯 영역을 초과하는지 여부를 확인하는 데 유용합니다.

문자열 내용의 오른쪽에 있는 필드는 다음과 같습니다.

- "참조 수": 이 필드는 GUIX Studio 프로젝트 내에서 특정 문자열 ID가 사용되는 빈도를 나타냅니다. 참조 횟수가 0이면 이 문자열은 사용되지 않을 수 있으며 사용자가 선택적으로 제거할 수 있습니다.
- 문자열 너비(픽셀)는 표시된 글꼴을 사용하여 문자열의 표시 너비를 나타냅니다.
- "참고" 필드는 각 문자열의 용도에 대한 정보를 추가할 수 있는 선택적 설명 필드입니다. 이러한 메모는 번역가가 정확하고 의미 있는 문자열 번역을 수행하는 데 도움이 되도록, 내보낸 XLIFF 문자열 데이터 파일에 포함되어 있습니다.

***문자열 테이블 편집기*** 대화 상자를 열 때마다 대화 상자 위쪽에 있는 문자열 추가 단추를 클릭하여 프로젝트에 추가 문자열을 추가할 수 있습니다. 먼저 문자열을 선택한 다음 대화 상자의 위쪽에 있는 문자열 삭제 단추를 클릭하여 프로젝트에서 사용되지 않는 문자열이나 사용하지 않는 문자열을 제거할 수 있습니다.

문자열 테이블 편집기 대화 상자를 사용하여 프로젝트에 새 문자열을 수동으로 추가하는 것 외에도 텍스트를 지원하는 위젯의 속성 보기의 "텍스트" 필드에 문자열 내용을 입력하기만 하면 새 문자열을 간접적으로 추가할 수 있습니다. 즉, 대상 뷰에서 새 위젯을 추가하거나 속성 보기에 텍스트 정보를 입력하는 경우 이러한 작업은 프로젝트 문자열 테이블에 자동으로 새 항목을 만듭니다.

## <a name="adding-language-translations"></a>언어 번역 추가

GUIX Studio 문자열 테이블 편집기는 개발자가 자신의 주 언어를 사용하여 애플리케이션을 만든 다음 언어 번역 전문가에게 보낼 표준 스키마 XML 또는 CSV 파일로 문자열 데이터를 내보낼 수 있도록 하는 언어 정의 워크플로를 지원합니다. 그러면 번역 파일이 개발자에게 반환되어 언어 번역을 다시 스튜디오 프로젝트로 가져올 수 있으므로 애플리케이션에 새 언어에 대한 지원이 추가됩니다.

이 기능은 문자열 테이블 편집기의 위쪽에 있는 내보내기(파일에 문자열 데이터 쓰기) 및 가져오기(번역된 문자열 읽기) 단추를 사용하여 호출됩니다. 내보내기 단추는 참조 언어 문자열을 포함하는 XLIFF 스키마 XML 또는 CSV 파일을 만드는 데 사용됩니다. 이 파일은 표준 XLIFF 또는 CSV 파일 형식을 지원하는 도구와 편집기를 사용하여 번역가가 활용할 수 있습니다.

번역 전문가가 새 문자열 번역과 함께 개발자에게 XLIFF 파일을 반환하면 가져오기 단추를 사용하여 이 XLIFF 또는 CSV 파일에서 데이터를 읽을 수 있습니다. XLIFF 또는 CSV 파일에 새 언어가 포함된 경우 새 언어가 프로젝트에 추가됩니다. XLIFF 파일에 기존 언어에 대한 새 문자열 데이터가 포함된 경우 이 새 데이터를 프로젝트로 가져옵니다. 참조 언어 문자열은 가져오기 작업으로 수정되지 않습니다.

내보내기 단추를 클릭하면 아래에 표시된 XLIFF/CSV 내보내기 제어 대화 상자가 표시됩니다.

![XLIFF/CSV 내보내기 제어 대화 상자의 스크린샷](./media/guix-studio/image54.jpg)

**그림 20**

원본 언어 및 대상 언어 필드는 XLIFF 또는 CSV 파일에 참조 언어 및 번역 언어로 기록될 문자열 테이블 열을 지정합니다. 원본 언어는 참조 문자열이고, 대상 언어는 번역가가 번역된 문자열 데이터를 제공하는 언어입니다.

XLIFF 버전 필드는 두 가지 기본 XLIFF 파일 형식 버전(버전 1.2 또는 버전 2.0 이상) 중 하나를 지정합니다. 이러한 XLIFF 파일 형식 표준은 호환되지 않으며, XLIFF 내보내기/가져오기 명령을 사용하기 전에 도구에서 사용하는 버전을 알아야 합니다. XLIFF 스키마 및 XLIFF 표준에 대한 자세한 내용은 다음을 참조하세요.

- 버전 1.2: [https://docs.oasis-open.org/xliff/xliff-core/xliff-core.html](https://docs.oasis-open.org/xliff/xliff-core/xliff-core.html)
- 버전 2.0: [https://docs.oasis-open.org/xliff/xliff-core/v2.0/os/xliff-core-v2.0os.pdf](https://docs.oasis-open.org/xliff/xliff-core/v2.0/os/xliff-core-v2.0-os.pdf)

출력 파일 이름 및 출력 경로 필드를 사용하여 출력 파일이 기록될 파일 이름과 위치를 지정할 수 있습니다. 파일 이름은 전적으로 사용자에게 제공되지만 내보낸 파일 내에 포함된 원본 및 대상 언어를 나타내는 이름을 사용하는 것이 좋습니다.
